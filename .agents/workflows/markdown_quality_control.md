# Markdown 品質管理ワークフロー

このワークフローは、Markdownファイルを新規作成、または修正する際に必ず参照し、構文エラー（Lintエラー）を未然に防ぐためのガイドラインです。

## 📋 Markdown スタイル・チェックリスト

### 1. 箇条書き記号の統一 (MD004)

* **ルール**: ファイル内で箇条書き（Unordered List）の記号は `*` （アスタリスク）に統一する。
* **理由**: `-` や `+` との混在を防ぎ、ファイル全体の一貫性を保つため。
* **修正方法**: 全箇条書きを `*` に変換する。

### 2. リストマーカー直後のスペース (MD030)

* **ルール**: `*` や `1.` の直後は **半角スペース1つのみ** とする。
* **理由**: 余分なインデントやスペースによるパースミスを防ぐため。
* **例**: `* 内容` (OK) / `* 内容` (NG)

### 3. 見出しとリストの周囲の空行 (MD022 / MD032)

* **ルール**: 見出し（`#`, `##`, `###`）の前と後、およびリスト構造の開始前と終了後には必ず **空行を1行以上** 入れる。
* **理由**: Markdown形式として正しく認識させ、見た目の可読性も高めるため。

### 4. 番号付きリストのプレフィックス (MD029)

* **ルール**: 親リストやインデントなしの番号付きリストは `1.`, `2.`, `3.` ... のように連番（Style: ordered）にする。
* **理由**: 番号が飛んだり、すべて `1.` になったりするのを防ぎ、構造を論理的に保つため。

### 5. 強調行を見出しとして使わない (MD036)

* **ルール**: `**Step 1**` のような強調のみで構成される行を独立して置かない。
* **理由**: それが見出しの意図であれば `### Step 1` のように見出しタグを使い、リストの一部であれば `* **Step 1**` のようにリスト項目にする。

## 🛠️ 使い方のフロー

1. **修正前/作成前チェック**
    * 編集対象のファイルが既に存在する場合、まず既存のリスト記号が `*` か `-` かを確認し、プロジェクト標準の `*` に合わせる準備をする。
2. **インライン修正**
    * 文中で箇条書きを追加する際は、前後に空行があることを確認してから追加する。
3. **最終仕上げ**
    * AIエージェント（私）は、ファイルを保存（`write_to_file` / `replace_file_content`）する直前に、このチェックリストに従って構文を最終確認する。
4. **Lintエラー発生時**
    * IDEからLintエラーの報告があった場合（MD0xx）、このワークフローの関連セクションを参照して修正を適用する。
